# Implementation Document Guide

## Summary
- **Goal:** Define a checklist-style implementation document format with enforceable AI rules.

## Purpose
- **Standardize implementation docs** into a single checklist table.
- **Make status explicit** with `todo`, `pending`, `done`, or `blocked` for each step.
- **Control AI edits** using the AI touchbase column.

## Scope
- **Applies to:** Implementation docs in `Implementation/`.
- **Does not apply to:** Design docs, runbooks, or general docs.

## Required Table Format
Use a single table with the columns below, in this order:
1) **Step ID**
2) **Step**
3) **Source design doc**
4) **Inputs**
5) **Outputs**
6) **Execution context**
7) **Execution location**
8) **Implementation status**
9) **AI touchbase**
10) **Remarks**

## Allowed Values
- **Implementation status:** `todo`, `pending`, `done`, or `blocked` only.
- **AI touchbase:** `locked`, `flexible`, or `must`.

## AI Touchbase Rules
- **locked:** Do not amend this step or its text. Preserve it exactly.
- **flexible:** Can edit wording or ordering to improve clarity.
- **must:** Must keep the step and ensure it is present.

## Writing Rules
- **One step per row**; no nested bullets.
- **Use imperative verbs** for steps (e.g., "Install", "Configure").
- **Keep inputs/outputs concrete** (paths, files, services).
- **Use repo-relative paths** for files.
- **Keep steps ordered** in the sequence they should be executed.
- **Execution location** should be explicit (e.g., VM, Local (repo), GitHub Actions runner).

## Enforcement
- **If a row is `locked`, do not change it.**
- **If a row is `must`, ensure it exists in the table.**
- **Only change rows marked `flexible` unless explicitly instructed.**
